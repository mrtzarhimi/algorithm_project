# پروژه مسیریابی شهری هوشمند
### طراحی و پیاده‌سازی هسته مسیریابی مبتنی بر گراف

---

# مقدمه و هدف پروژه

## هدف

هدف این پروژه طراحی و پیاده‌سازی هستهٔ مسیریابی برای یک سیستم شهری هوشمند (مشابه سامانه‌های درخواست خودرو) است.

در این پروژه، نقشهٔ شهر به‌صورت یک **گراف جهت‌دار و وزن‌دار** مدل شده است که:

- هر **گره (Vertex)** نمایانگر یک تقاطع است.
- هر **یال (Edge)** نمایانگر یک خیابان بین دو تقاطع است.

هزینهٔ عبور از هر خیابان بر اساس چهار مؤلفه محاسبه می‌شود:

- مسافت (Distance)
- ضریب ترافیک (Traffic Coefficient)
- ضریب شرایط جاده و آب‌وهوا (Weather/Road Coefficient)
- تأخیر تقاطع مقصد (Intersection Delay)

پروژه در چهار فاز پیاده‌سازی شده است:

1. مدل‌سازی گراف و خواندن نقشه از فایل ورودی
2. یافتن کوتاه‌ترین مسیر با الگوریتم دایجسترا
3. پشتیبانی از وزن‌های منفی و تشخیص دور منفی با الگوریتم بلمن-فورد
4. پیشنهاد مسیر چندمقصدی با روش تقریبی نزدیک‌ترین همسایه (Nearest Neighbor)

---

# فاز اول: مدل‌سازی گراف و خواندن نقشه

## ساختار داده

برای نمایش گراف در حافظه از **فهرست مجاورت (Adjacency List)** استفاده شده است و از ماتریس مجاورت استفاده نشده است.

دلیل این انتخاب آن است که در نقشه‌های شهری، تعداد خیابان‌های خروجی از هر تقاطع معمولاً محدود است؛ بنابراین استفاده از فهرست مجاورت:

- حافظهٔ کمتری مصرف می‌کند.
- پیمایش همسایه‌های هر گره را سریع‌تر انجام می‌دهد.
- برای گراف‌های تنک (Sparse Graphs) مناسب‌تر است.

ساختارهای اصلی داده به صورت زیر تعریف شده‌اند:

```cpp
struct Edge {
    int to;
    double distance;
    double traffic;
    double weather;
};

int V, E;

string vertexName[MAXV];
double delay[MAXV];

vector<Edge> adjacency[MAXV];
```

در این طراحی، هر گره با یک اندیس صحیح مشخص می‌شود و تابع `findVertexIndex` نام هر تقاطع را به اندیس متناظر آن تبدیل می‌کند.

همچنین آرایهٔ `delay` مقدار تأخیر هر تقاطع را نگهداری می‌کند و هنگام محاسبهٔ هزینهٔ ورود به آن گره مورد استفاده قرار می‌گیرد.

## تابع هزینه

هزینهٔ عبور از هر یال مطابق رابطهٔ زیر محاسبه می‌شود:

```cpp
double getEdgeCost(Edge e) {
    return e.distance * e.traffic * e.weather + delay[e.to];
}
```

در این رابطه:

- مسافت، هزینهٔ پایه است.
- ضرایب ترافیک و شرایط آب‌وهوا به‌صورت ضربی بر هزینه اثر می‌گذارند.
- تأخیر تقاطع مقصد به‌صورت جمعی به هزینه اضافه می‌شود.

## خواندن فایل ورودی

تابع `readMapFile` مسئول خواندن فایل نقشه و ساخت گراف در حافظه است.

ترتیب خواندن اطلاعات مطابق قالب ورودی به صورت زیر است:

1. خواندن تعداد گره‌ها (`V`) و تعداد یال‌ها (`E`)
2. خواندن نام و مقدار تأخیر هر گره
3. خواندن اطلاعات هر یال شامل:
   - گره مبدأ
   - گره مقصد
   - مسافت
   - ضریب ترافیک
   - ضریب آب‌وهوا
4. خواندن گرهٔ مبدأ و مقصد برای اجرای الگوریتم‌های مسیریابی

در صورتی که نام هر گره در فایل ورودی معتبر نباشد، برنامه با پیام خطا متوقف می‌شود.

تابع `printGraph` نیز برای نمایش گراف ساخته‌شده و بررسی صحت ورودی استفاده شده است.

## تحلیل پیچیدگی

ساخت گراف تنها یک بار تمام گره‌ها و یال‌ها را پیمایش می‌کند؛ بنابراین:

- **پیچیدگی زمانی:** `O(V + E)`
- **پیچیدگی فضایی:** `O(V + E)`

### محدودیت‌های طراحی

- اندازهٔ گراف با ثابت `MAXV = 100` محدود شده است.
- تابع `findVertexIndex` از جست‌وجوی خطی استفاده می‌کند و دارای پیچیدگی `O(V)` است.
- برای گراف‌های بزرگ‌تر، استفاده از `unordered_map` می‌تواند زمان جست‌وجوی نام گره را به حدود `O(1)` کاهش دهد.

# فاز دوم: کوتاه‌ترین مسیر با الگوریتم دایجسترا

## معرفی

در این فاز، کوتاه‌ترین مسیر بین یک گره مبدأ و سایر گره‌ها با استفاده از الگوریتم **دایجسترا (Dijkstra)** محاسبه می‌شود. این الگوریتم تنها زمانی معتبر است که تمام وزن‌های گراف نامنفی باشند؛ در این پروژه نیز هزینهٔ هر یال توسط تابع `getEdgeCost` محاسبه شده و به‌عنوان وزن یال مورد استفاده قرار می‌گیرد.

تابع اصلی این بخش، `dijkstra` در فایل `phase2.cpp` است.

---

## نحوهٔ پیاده‌سازی

در این پیاده‌سازی، از نسخهٔ سادهٔ الگوریتم دایجسترا استفاده شده است. در هر مرحله، گره بازدیدنشده‌ای که کمترین فاصلهٔ موقت را دارد با یک پیمایش خطی انتخاب می‌شود و سپس تمام یال‌های خروجی آن بررسی (Relax) می‌شوند.

```cpp
for (int count = 0; count < V; count++) {

    int u = -1;
    double bestDist = INF;

    for (int i = 0; i < V; i++) {
        if (!visited[i] && dist[i] < bestDist) {
            bestDist = dist[i];
            u = i;
        }
    }

    if (u == -1)
        break;

    visited[u] = true;

    for (Edge e : adjacency[u]) {

        double newDist = dist[u] + getEdgeCost(e);

        if (newDist < dist[e.to]) {
            dist[e.to] = newDist;
            parent[e.to] = u;
        }
    }
}
```

در این الگوریتم:

- آرایهٔ `dist` کمترین هزینهٔ شناخته‌شده تا هر گره را نگهداری می‌کند.
- آرایهٔ `visited` مشخص می‌کند که کدام گره‌ها نهایی شده‌اند.
- آرایهٔ `parent` برای بازسازی مسیر نهایی استفاده می‌شود.

---

## بازسازی مسیر

پس از پایان الگوریتم، مسیر کوتاه‌ترین مسیر توسط تابع `printPath` بازسازی می‌شود.

این تابع از گرهٔ مقصد شروع کرده و با دنبال کردن آرایهٔ `parent` تا رسیدن به مبدأ حرکت می‌کند. سپس مسیر به‌دست‌آمده معکوس شده و به‌ترتیب صحیح نمایش داده می‌شود.

این روش همان روش استاندارد بازسازی مسیر در الگوریتم دایجسترا است.

---

## تحلیل پیچیدگی

در این پروژه، برای انتخاب گره با کمترین فاصله از **پیمایش خطی** استفاده شده و از صف اولویت (Priority Queue) استفاده نشده است؛ بنابراین:

- انتخاب گره با کمترین فاصله: `O(V)`
- این عملیات برای تمام گره‌ها تکرار می‌شود.
- هر یال نیز حداکثر یک بار بررسی (Relax) می‌شود.

در نتیجه:

- **پیچیدگی زمانی:** `O(V² + E)` که برای گراف‌های تنک معمولاً به صورت `O(V²)` بیان می‌شود.
- **پیچیدگی فضایی:** `O(V + E)`

---

## محدودیت

این پیاده‌سازی تنها زمانی صحیح عمل می‌کند که تمام وزن‌های گراف نامنفی باشند. در صورت وجود حتی یک یال با وزن منفی، تضمین بهینگی الگوریتم از بین می‌رود و ممکن است مسیر نادرستی محاسبه شود.

به همین دلیل، در فاز بعدی از الگوریتم **بلمن-فورد** برای پشتیبانی از وزن‌های منفی استفاده شده است.

# فاز سوم: کوتاه‌ترین مسیر با الگوریتم بلمن-فورد

## معرفی

در این فاز، الگوریتم **بلمن-فورد (Bellman-Ford)** برای محاسبهٔ کوتاه‌ترین مسیر در گراف‌هایی که ممکن است دارای **وزن منفی** باشند، پیاده‌سازی شده است.

برخلاف الگوریتم دایجسترا، بلمن-فورد قادر است علاوه بر محاسبهٔ کوتاه‌ترین مسیر، وجود **دور منفی (Negative Cycle)** را نیز تشخیص دهد. در صورت وجود چنین دوری، مفهوم کوتاه‌ترین مسیر تعریف‌نشده است؛ زیرا می‌توان با پیمایش مکرر آن دور، هزینهٔ مسیر را به‌صورت نامحدود کاهش داد.

تابع اصلی این بخش، `bellmanFord` در فایل `phase3.cpp` است.

---

## نحوهٔ پیاده‌سازی

الگوریتم بلمن-فورد در دو مرحله اجرا می‌شود.

### مرحلهٔ اول: Relax کردن یال‌ها

در این مرحله، تمام یال‌های گراف به تعداد `V - 1` بار بررسی می‌شوند. در هر بار، اگر عبور از یک یال باعث کاهش هزینهٔ رسیدن به گرهٔ مقصد شود، مقدار فاصله و گرهٔ والد به‌روزرسانی می‌شود.

```cpp
for (int i = 0; i < V - 1; i++) {

    for (int u = 0; u < V; u++) {

        if (dist[u] == INF)
            continue;

        for (Edge e : adjacency[u]) {

            double newDist = dist[u] + getEdgeCost(e);

            if (newDist < dist[e.to]) {
                dist[e.to] = newDist;
                parent[e.to] = u;
            }
        }
    }
}
```

---

### مرحلهٔ دوم: تشخیص دور منفی

پس از پایان مراحل Relax، تمام یال‌ها یک بار دیگر بررسی می‌شوند.

اگر هنوز بتوان فاصلهٔ یکی از گره‌ها را کاهش داد، وجود یک دور منفی در گراف قطعی است و الگوریتم پایان می‌یابد.

```cpp
for (int u = 0; u < V; u++) {

    if (dist[u] == INF)
        continue;

    for (Edge e : adjacency[u]) {

        double newDist = dist[u] + getEdgeCost(e);

        if (newDist < dist[e.to]) {
            return false;
        }
    }
}
```

در این پیاده‌سازی، مقدار `false` نشان‌دهندهٔ وجود دور منفی است و در این حالت برنامه از نمایش مسیر خودداری کرده و پیام مناسب را نمایش می‌دهد.

---

## بازسازی مسیر

در صورتی که دور منفی وجود نداشته باشد، مسیر کوتاه‌ترین مسیر دقیقاً مشابه فاز قبل، با استفاده از آرایهٔ `parent` توسط تابع `printPath` بازسازی و نمایش داده می‌شود.

---

## تحلیل پیچیدگی

در هر یک از `V - 1` مرحله، تمام یال‌های گراف بررسی می‌شوند و در انتها نیز یک پیمایش دیگر برای تشخیص دور منفی انجام می‌شود.

بنابراین:

- **پیچیدگی زمانی:** `O(V × E)`
- **پیچیدگی فضایی:** `O(V)` (به‌جز فضای موردنیاز برای ذخیرهٔ گراف)

---

## مقایسهٔ دایجسترا و بلمن-فورد

| ویژگی | دایجسترا | بلمن-فورد |
|--------|----------|-----------|
| پشتیبانی از وزن منفی | ✗ | ✓ |
| تشخیص دور منفی | ✗ | ✓ |
| سرعت اجرا | بیشتر | کمتر |
| کاربرد در پروژه | کوتاه‌ترین مسیر برای وزن‌های نامنفی | کوتاه‌ترین مسیر با وزن منفی و تشخیص دور منفی |

در نتیجه، اگر تمام وزن‌های گراف نامنفی باشند، استفاده از دایجسترا به دلیل سرعت بیشتر مناسب‌تر است. در مقابل، زمانی که امکان وجود وزن منفی یا نیاز به تشخیص دور منفی وجود داشته باشد، الگوریتم بلمن-فورد انتخاب صحیح خواهد بود.

# فاز چهارم (امتیازی): مسیر چندمقصدی با الگوریتم نزدیک‌ترین همسایه

## معرفی

در این فاز، مسئلهٔ مسیریابی از حالت تک‌مقصدی به **چندمقصدی** توسعه داده شده است.

در این سناریو، پیک از یک مبدأ حرکت می‌کند، باید به مجموعه‌ای از مقصدهای مشخص مراجعه کند و در پایان به نقطهٔ شروع بازگردد. این مسئله نسخه‌ای ساده‌شده از **مسئلهٔ فروشندهٔ دوره‌گرد (Travelling Salesman Problem - TSP)** است.

از آنجا که مسئلهٔ TSP از مسائل **NP-Hard** محسوب می‌شود و یافتن جواب بهینه برای تعداد زیاد مقصدها هزینهٔ محاسباتی بسیار بالایی دارد، در این پروژه از روش تقریبی **نزدیک‌ترین همسایه (Nearest Neighbor)** استفاده شده است.

---

## نحوهٔ پیاده‌سازی

در فایل `phase4.cpp`، ابتدا تابع `shortestPath` برای محاسبهٔ کوتاه‌ترین مسیر بین دو گره پیاده‌سازی شده است. این تابع بر پایهٔ الگوریتم دایجسترا عمل می‌کند و هزینهٔ کوتاه‌ترین مسیر بین دو نقطه را محاسبه می‌کند.

سپس تابع `runPhase4` فرآیند انتخاب مقصدها را به صورت زیر انجام می‌دهد:

1. حرکت از گرهٔ مبدأ.
2. محاسبهٔ هزینهٔ کوتاه‌ترین مسیر تا تمام مقصدهای بازدیدنشده.
3. انتخاب نزدیک‌ترین مقصد.
4. اضافه کردن آن به مسیر نهایی.
5. تکرار مراحل بالا تا زمانی که تمام مقصدها بازدید شوند.
6. محاسبهٔ هزینهٔ بازگشت از آخرین مقصد به مبدأ.

در نتیجه، مسیر نهایی به‌صورت حریصانه (Greedy) ساخته می‌شود و در هر مرحله بهترین انتخاب محلی انجام می‌شود.

---

## محدودیت روش

الگوریتم نزدیک‌ترین همسایه یک روش تقریبی است و **همیشه مسیر بهینه را تضمین نمی‌کند**. ممکن است انتخاب بهترین مقصد در یک مرحله باعث ایجاد مسیری با هزینهٔ بیشتر در ادامه شود.

همچنین، تابع `shortestPath` در این فاز بر پایهٔ الگوریتم دایجسترا نوشته شده است؛ بنابراین این بخش تنها برای گراف‌هایی که همهٔ وزن‌های آن نامنفی باشند معتبر است. در صورت وجود وزن منفی، برنامه به‌صورت خودکار از بلمن-فورد استفاده نمی‌کند و ممکن است نتیجهٔ نادرستی تولید شود.

---

## تحلیل پیچیدگی

فرض کنیم تعداد مقصدهای موردنظر برابر `K` باشد.

در هر مرحله، برای تمام مقصدهای بازدیدنشده کوتاه‌ترین مسیر محاسبه می‌شود. در بدترین حالت، این محاسبات تقریباً `K²` بار انجام می‌شوند و هر بار نیز هزینهٔ اجرای الگوریتم کوتاه‌ترین مسیر پرداخت می‌شود.

بنابراین پیچیدگی زمانی این فاز به صورت زیر بیان می‌شود:

- **پیچیدگی زمانی:** `O(K² × SP)`

که در آن `SP` هزینهٔ اجرای الگوریتم کوتاه‌ترین مسیر است:

- اگر از دایجسترا استفاده شود، `SP` برابر پیچیدگی الگوریتم دایجسترا است.
- اگر از بلمن-فورد استفاده شود، `SP` برابر پیچیدگی الگوریتم بلمن-فورد خواهد بود.

فضای موردنیاز نیز شامل فضای ذخیره‌سازی گراف و اطلاعات مقصدها است:

- **پیچیدگی فضایی:** `O(V + K)`

---

## جمع‌بندی

روش نزدیک‌ترین همسایه با وجود آنکه تضمینی برای یافتن مسیر بهینه ندارد، به دلیل سادگی پیاده‌سازی و سرعت مناسب، انتخاب مناسبی برای مسائل مسیریابی چندمقصدی با تعداد مقصدهای محدود است.

# نتیجه‌گیری

در این پروژه، یک هستهٔ مسیریابی برای یک سیستم شهری هوشمند بر پایهٔ گراف جهت‌دار و وزن‌دار طراحی و پیاده‌سازی شد. ابتدا ساختار گراف با استفاده از فهرست مجاورت پیاده‌سازی شد و تابع هزینهٔ هر یال بر اساس مسافت، ترافیک، شرایط جاده و آب‌وهوا و تأخیر تقاطع مقصد تعریف گردید.

در ادامه، الگوریتم دایجسترا برای محاسبهٔ کوتاه‌ترین مسیر در گراف‌های با وزن نامنفی پیاده‌سازی شد. سپس با استفاده از الگوریتم بلمن-فورد، امکان پشتیبانی از وزن‌های منفی و تشخیص دور منفی به پروژه اضافه شد. در بخش امتیازی نیز با بهره‌گیری از روش تقریبی نزدیک‌ترین همسایه، مسئلهٔ مسیریابی چندمقصدی پیاده‌سازی شد.

اگرچه پیاده‌سازی حاضر تمامی اهداف اصلی پروژه را پوشش می‌دهد، امکان توسعهٔ آن نیز وجود دارد. از جمله بهبودهای آینده می‌توان به استفاده از `unordered_map` برای جست‌وجوی سریع‌تر گره‌ها، استفاده از صف اولویت (Priority Queue) برای بهینه‌سازی الگوریتم دایجسترا، حذف محدودیت ثابت `MAXV` با استفاده از ساختارهای دادهٔ پویا، و استفاده از الگوریتم‌های پیشرفته‌تر برای حل تقریبی مسئلهٔ فروشندهٔ دوره‌گرد اشاره کرد.

در مجموع، این پروژه علاوه بر پیاده‌سازی عملی چند الگوریتم مهم گراف، تفاوت کاربردها، مزایا و محدودیت‌های آن‌ها را در یک مسئلهٔ واقعی مسیریابی شهری نشان می‌دهد.
